# 9. Absolute Rule: No Data Leakage

This is the most important requirement.

Every feature must be computed using only information that would have been available at the decision timestamp.

Forbidden:

* Using future candles to create past features
* Using future-confirmed swing points as real-time signals
* Using same-candle close for an entry that supposedly happened before that close
* Fitting scalers on the full dataset
* Tuning thresholds on the final test set
* Selecting features using validation/test information incorrectly
* Using labels or future returns in feature generation
* Computing rolling features with centered windows
* Backtesting with impossible intrabar assumptions
* Using ZigZag/fractals in a future-leaking way
* Resampling higher timeframe data and forward-filling unfinished higher timeframe candles
* Using future FVG/order block/sweep information in past decisions

Required:

* Features must be point-in-time safe
* Signal generated after candle close
* Trade entry no earlier than next candle open or realistic simulated execution price
* Scalers fit only on train data
* Validation/test transformed only with train-fitted scalers
* Chronological splits
* Walk-forward validation
* Untouched final test set
* Explicit leakage tests

Implement leakage checks in:

```text
src/validation/leakage_checks.py
```

Every major notebook must include a leakage-safety section.
